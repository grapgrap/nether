# loom

loom은 컨셉 기반 설계 시스템이다. 소프트웨어의 설계 의도를 문서화하고, 결정의 맥락을 보존한다.

## 원칙

- **What/Why와 How를 분리한다.** concept은 "무엇을, 왜"를 정의하고, design은 "어떻게"를 다룬다.
- **컨셉은 사용자가 인지하는 행동 단위다.** 코드나 모듈이 아닌, 사용자 관점의 추상적 개념을 기준으로 설계한다.
- **결정의 맥락을 보존한다.** 모든 선택은 decision으로 기록하여 "왜 이렇게 결정했는가"를 추적한다.
- **워크플로를 강제하지 않는다.** loom은 사용자의 워크플로에 녹아드는 도구이며, 커맨드 간 순서를 지시하지 않는다. 진입점을 분산하여 유연성을 확보한다.

## 컨셉 목록

| 컨셉 | 설명 |
|------|------|
| [[concept]] | 소프트웨어 설계의 기본 단위 (What/Why) |
| [[design]] | 컨셉의 구현 방법 (How), concept과 1:1 대응 |
| [[decision]] | 모든 단계를 관통하며 선택의 맥락을 기록하는 횡단 컨셉 |
| [[plan]] | 설계 또는 외부 스펙을 실행 가능한 작업 단위로 분해. 구현 시 코파일럿 역할 수행 |
| [[index]] | 각 scope의 진입점이자 자기설명 문서 |

## 관계 구조

```
concept (핵심)
├── design (구현 방법, 1:1 대응)
│   └── plan (작업 분해, .gitignore 대상 파일, 구현 시 코파일럿, 역방향 시 design을 생성)
├── decision (모든 단계를 관통하는 선택 추적)
└── index (scope의 진입점, 경계와 지도)
```

## 핵심 흐름

1. **specify**: 요구사항 → concept 도출 (What/Why)
2. **design**: concept → design 작성 (How)
3. **plan**: design 또는 외부 스펙 → 실행 가능한 작업 계획
   - 정방향: design이 입력, 작업 분해에 집중
   - 역방향: 외부 스펙이 입력, concept/design/decision을 함께 생성
4. **decision**: 각 단계의 선택을 기록
