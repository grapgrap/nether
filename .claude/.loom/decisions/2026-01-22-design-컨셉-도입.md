# 2026-01-22-design-컨셉-도입

## 맥락

loom 시스템에서 `/loom.specify`는 What/Why에 집중하며, How를 다루지 않았다.
사용자가 구현 단계로 넘어갈 때 사용할 명령어가 필요했다.

## 결정 내용

### 1. `/loom.design` 명령어 도입

컨셉의 구현 방법(How)을 다루는 새 명령어를 추가한다.
상세 구현이 아닌 **모델링, 데이터 흐름, 비즈니스 로직** 수준을 다룬다.

### 2. 명칭: plan → design

"plan"은 미래 행동의 순서를 암시하지만, 실제 내용은 구조와 규칙에 가깝다.
"design"이 concept과 자연스러운 짝을 이룬다.

### 3. concept과 1:1 대응

design과 concept은 1:1로 대응한다.
컨셉의 계층 구조(하위/관련 컨셉)가 design의 범위를 자연스럽게 결정한다.

### 4. 태스크 목록 제외

design 문서에서 태스크 목록은 제외한다.
태스크는 별도 시스템에서 관리한다.

### 5. specify → design 전환 방식

specify에서 "How 시그널"을 감지하면 design으로 전환을 안내한다.
별도 인덱스 문서 없이 자연스러운 대화 흐름으로 연결한다.

### 제약 조건

- loom 시스템의 기존 구조(scope, concept, decision)와 일관성 유지
- 문서 예시 최소화, 설명 자체로 의미 전달

### 전제 조건

- 모든 컨셉은 하위 또는 관련 컨셉과의 관계를 가진다
- design은 반드시 대응하는 concept이 있어야 한다

## 대안

### N:M 관계

- 설명: design과 concept이 다대다 관계를 가짐
- 트레이드오프: 관계 관리가 복잡해지고, 네이밍/식별이 어려움. 컨셉의 계층 구조를 활용하면 1:1로 충분함

### plan 명칭 유지

- 설명: `/loom.plan`으로 명명
- 트레이드오프: "plan"은 시간적 순서를 암시하여 실제 내용(구조, 규칙)과 맞지 않음

### 워크플로우 인덱스 문서

- 설명: `.loom/workflow.md`로 specify → design 흐름을 문서화
- 트레이드오프: 추후 필요시 추가 가능. 현재는 대화 흐름으로 충분
