# 2026-02-12-plan-컨셉-도입

## 맥락

loom 시스템에서 specify(What/Why) → design(How)까지 도출된 설계를 실제 구현으로 연결하는 단계가 없었다.
design은 모델링·흐름 수준의 설계를 다루지만, 이를 실행 가능한 작업으로 분해하는 것은 설계자의 암묵적 판단에 의존하고 있었다.

## 결정 내용

### 1. `/loom.plan` 명시적 커맨드 도입

design을 입력으로 받아 구현 계획을 세우는 전용 커맨드를 추가한다.
내부적으로 Claude의 plan mode로 동작하되, loom 맥락(concept, design, decision)을 자동으로 주입한다.

### 2. 영속 문서 불필요

plan은 concept이나 design과 달리 파일로 영속되지 않는다.
대화 컨텍스트 안에서 구현 중 참조 가능하면 충분하다.
사용자 요청 시 외부 목적지(티켓 관리 서비스 등)에 출력할 수 있다.

### 3. 저장 위치 무관심

plan은 출력 목적지에 대해 의견을 갖지 않는다.
기본은 대화 컨텍스트이며, 사용자가 지정하면 외부 서비스에 기록한다.

### 제약 조건

- loom 시스템의 기존 구조(scope, concept, design, decision)와 일관성 유지
- Claude의 기존 plan mode 기능을 활용하여 중복 구현 방지

### 전제 조건

- plan은 반드시 대응하는 design이 있어야 한다
- plan mode가 loom 문서를 참조할 수 있으려면 커맨드가 맥락을 주입해야 한다

## 대안

### design 커맨드 완료 후 plan mode 자동 전환

- 설명: `/loom.design` 완료 시 별도 커맨드 없이 바로 plan mode로 진입
- 트레이드오프: loom 맥락(concept, design, decision) 자동 주입이 보장되지 않는다. design 후 바로 구현하지 않는 경우도 있어 항상 전환하는 것은 부자연스럽다

### plan을 design의 하위 섹션으로 포함

- 설명: design 문서에 "구현 계획" 섹션을 추가
- 트레이드오프: design의 경계("상세 코드 구현은 다루지 않는다")를 침범한다. 영속 문서에 일회성 작업 목록이 섞이게 된다
