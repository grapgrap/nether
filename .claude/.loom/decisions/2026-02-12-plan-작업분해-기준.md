# 2026-02-12-plan-작업분해-기준

## 맥락

plan 컨셉 도입 후, 작업 분해의 구체적인 기준과 의존성 표현 방식을 정의할 필요가 있었다.
"작업 단위"의 크기, 분해의 입력, 의존성 정보의 표현 형태가 미정이었다.

## 결정 내용

### 1. 작업 분해 기준: 코드 초안 작성 가능 수준

작업 하나의 크기는 **구현자가 코드 초안을 바로 작성할 수 있을 정도**로 한다.
"데이터 모델 구축" 같은 추상적 수준이 아니라, "Post 엔티티의 Zod 스키마와 타입 정의" 같은 구체적 수준이다.

### 2. 분해의 입력은 design 문서의 각 요소

design이 다루는 모델링, 데이터 흐름, 비즈니스 로직 각 요소가 작업 분해의 자연스러운 입력이 된다.

### 3. 본문은 단계적 나열, 병렬 정보는 참조 섹션에 분리

plan의 출력 구조:
- **본문**: 의존성에 따른 단계적 작업 목록
- **참조 섹션**: 어떤 작업들이 병렬 진행 가능한지 별도 기술

### 4. 병렬성은 연관관계의 강도를 드러내는 장치

병렬 가능 여부를 표기하는 목적은 동시 실행을 지시하는 것이 아니다.
작업 간 연관관계가 약하다는 설계적 정보를 제공하는 것이며, 실제 병렬 수행 여부는 작업자가 판단한다.

### 제약 조건

- design 문서가 모델링·데이터 흐름·비즈니스 로직 수준으로 충분히 구체적이어야 코드 초안 수준의 분해가 가능하다

### 전제 조건

- plan은 반드시 대응하는 design이 있어야 한다
- design의 구조가 작업 분해의 축이 된다

## 대안

### 작업 단위를 모듈/파일 수준으로 고정

- 설명: 작업 하나 = 파일 하나 또는 모듈 하나로 기계적으로 대응
- 트레이드오프: 설계 구조와 파일 구조가 항상 일치하지 않는다. 하나의 설계 요소가 여러 파일에 걸치거나, 하나의 파일이 여러 설계 요소를 포함할 수 있다

### 병렬 가능 작업을 stage/phase로 그루핑

- 설명: Stage 1 (병렬), Stage 2 (병렬) 식으로 단계별 그룹을 만들어 본문에 포함
- 트레이드오프: 본문의 단계적 흐름이 그루핑에 의해 왜곡될 수 있다. 병렬성은 부가 정보이므로 참조 섹션에 분리하는 것이 관심사 분리에 부합한다
