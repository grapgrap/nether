# Index

각 `.loom/` 디렉토리의 진입점으로서, scope의 경계를 정의하고 컨셉 구조를 안내하는 자기설명 문서.

## 목적

loom 커맨드와 사용자가 작업을 시작할 때, 해당 scope의 맥락을 빠르게 파악할 수 있게 한다. `.loom/` 디렉토리의 물리적 위치가 scope의 경계를 정의하고, index는 그 안의 목적, 원칙, 컨셉 구조를 제공한다.

### 핵심 시나리오

- loom 커맨드 실행 시 index를 먼저 읽으면, 현재 scope의 맥락과 컨셉 구조를 파악할 수 있다
- 사용자의 요구사항과 index의 구조를 함께 보면, 어떤 컨셉 문서를 먼저 읽어야 할지 판단할 수 있다
- `.loom/` 디렉토리의 위치가 scope를 결정하므로, 별도의 scope 설정 없이 디렉토리 구조만으로 경계가 드러난다

## 동작 원리

### 주요 개념

- **자기설명**: scope의 목적과 원칙을 기술. 시스템 scope(`.claude/.loom/`)에서는 loom 시스템의 원칙을, 도메인 scope(`packages/{서비스명}/.loom/`)에서는 도메인 맥락을 담는다
- **경계**: `.loom/` 디렉토리의 물리적 위치가 scope를 결정한다. 각 scope는 독립적이며, scope 간 상속이나 공유는 없다
- **컨셉 목록**: scope 내에 존재하는 모든 컨셉
- **관계 구조**: 컨셉 간의 연결 (조합, 동기화, 의존)

### 핵심 동작

- **안내**: loom 커맨드가 index를 읽고 scope의 맥락과 컨셉 구조를 파악
- **등록**: 새 컨셉 도출 시 index에 추가
- **갱신**: 기존 컨셉 간 새 관계 설정 시 index 업데이트

### Scope 결정 방식

loom 커맨드가 동작할 scope를 결정하는 방법:

1. **암시적 추론**: 대화 맥락에서 자동 추론
   - 명시적 서비스명 ("blog", "meeting")
   - 도메인 키워드 ("워크플로", "화상회의", "인증")

2. **명시적 지정**: loom 커맨드에 서비스명을 인자로 전달하여 직접 설정

**탐색 흐름**:

1. 대화 맥락에서 키워드 추출
2. `packages/` 디렉토리에서 관련 서비스 탐색
3. 명확하면 해당 서비스 사용, 불명확하면 사용자 확인

## 경계

- 컨셉의 상세 내용 (개별 컨셉 문서에서)

- scope 간 상속/공유 미지원
- 물리적 디렉토리 구조에 종속
